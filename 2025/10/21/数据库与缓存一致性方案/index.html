MySql与Redis双写一致性方案

Created2025-10-21|Updated2025-10-22
|Post Views:

背景

在实际开发中，经常需要使用redis做cache，提高读性能。而引入缓存后，需要注意缓存与数据库一致性的问题。

CacheAside 旁路缓存

Redis 和 Mysql 数据发生不一致的本质：

更新数据，设计到两个代码操作：

  1. 更新数据库
  2. 更新Redis

发生不一致的本质：这两个操作不是原子性的，因此在并发下，多线程的执行顺序导致数据库不一致。

如：

为什么要先更新数据库，在更新缓存

先更新缓存，在更新数据库的情况：

线程A, 线程B都执行写操作。

可能的执行顺序：

  1. 线程A更新缓存值为A
  2. 线程B更新缓存值为B
  3. 线程B更新数据库B
  4. 线程A更新数据库A

此时，缓存值为B 数据库值为A 不一致。

为什么是删除缓存，而不是更新缓存为新值

线程A, 线程B都执行写操作。

  1. 线程A更新数据库A
  2. 线程B更新数据库B
  3. 线程B更新缓存B
  4. 线程A更新缓存A

此时，数据库B，缓存A 不一致

因此本质就是更新数据库，更新缓存这两个操作不是原子的，因此在此之间能发生线程安全问题。

那先更新数据库，在删除缓存，就能保证数据库一致了吗？

不不不，还有一种情况，那就是更新数据库后，还没删除缓存，这个时间段其他线程读缓存读到的是旧值，数据不一致。

但是，这种情况与上述不同的是，由于代码执行到删除就会一致，不一致的时间很短，其余的是，如果其他线程没有再发生写入，那么会一直不一致。

与CacheAside一起使用的策略

借助其他手段，更好的保证一致性

延时双删

解决痛点：

我们知道生产环境一般来说都是有主从节点的，采用的是读写分离的策略（从库读，主库写），那主从同步的这一段时间里，主从数据是不一致的。

知道这个背景，那当我们主线程（写的那个线程）更新完数据库（主库），删除缓存后，其他线程读数据库（从库未同步），读到的是旧值，写入缓存的也是旧值，这就发生了数据库，缓存不一致的现象！

因此我们采用延时双删的策略，目的是为了允许主从同步的时间

重试机制

删除缓存可能会失败，因此需要设定重试机制来保证缓存删除成功。

不能一直重试，会严重占用系统资源

异步，利用Canal监听数据库binlog，异步删除缓存

好处：

  1. 减少代码倾入，直接更新数据库，不用再写缓存更新的代码
  2. 延时双删的时间点设置很关键，太短没用，太慢数据库不一致的时间又有点长。
  3. 避免删除失败阻塞主线程
