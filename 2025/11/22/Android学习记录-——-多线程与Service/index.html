异步机制 —— Handler

消息队列机制，有对应的生产者和消费者，实际消费者在主线程

  1. Message: 线程间传递的消息

  2. Handler：消息的收发

  3. 发送：handler.sendMessage(msg), post

  4. 处理：handleMessage(msg: Message)

  5. MessageQueue：消息队列，主要用于存放所有通过Handler发送的消息，每个线程都有一个对应的MessageQueue对象

  6. Looper：每个线程的MessageQueue管家，可以理解为轮询访问MessageQueue发现新的待消费消息，调用handleMessage消费。

总体可以理解为：子线程生产，主线程消费。

Service

Service 是 Android 中用于执行长时间运行任务的组件，可与 Activity 解耦，在应用处于后台甚至没有界面时继续工作。

如现在有个下载功能，由Activity决定合适下载，Service提供下载服务

写法：

  1. 定义binder类实现对应功能
  2. 重写onBind方法
  3. Activity与Service绑定：建立连接，bindService

生命周期

方法 时机 说明
onCreate() 首次创建 初始化资源，只调用一次
onStartCommand() startService() 后 每次 startService() 都会触发
onBind() bindService() 时 返回 Binder，与客户端建立通信
onDestroy() 服务停止/回收 释放资源

主要关注四个方法：onCreate(), onStartCommand(), onBind(), onDestroy()

启动阶段：当调用了Context的startService()时，会启动对应service，此时，onCreate()会先于onStartCommand()执行。

绑定阶段：context调用bindService()时，会回调onBind()，在此基础上再调用 unbindService()，会调用onDestroy()

终止阶段：当调用了Context的startService()时（Service存活），再调用stopService()或stopSelf()，或触发系统回收时，服务终止，调用onDestroy()

​ .setLargeIcon(BitmapFactory.decodeResource(resources, R.drawable.large_icon))这里的图片不能随便定义，因为通知的小图标 (setSmallIcon) 和大图标 (setLargeIcon) 都有尺寸与格式规范。如果使用过大的位图或带透明背景的复杂图，系统会在渲染时裁剪、缩放甚至直接丢弃，从而导致通知无法正常显示或图标变成空白。

如何排查：Notification: small icon must be set 或 Bad notification posted
