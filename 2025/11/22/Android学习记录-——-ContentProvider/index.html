运行时权限

运行时权限，不需要一次性申请所有权限，而是在软件使用过程中再对某个权限申请授权

代码实例

解读：

  1. ContextCompat.checkSelfPermission(this, Manifest.permission.READ_CONTACTS)检查是否有该权限， 与PackageManager.PERMISSION_GRANTED作对比，相等说明已授权，否则调用ActivityCompat.requestPermissions()申请获得权限。
  2. 调用ActivityCompat.requestPermissions()后，会回调onRequestPermissionsResult()方法。

基本用法

ContentResolver

使用场景：用于访问contentProvider数据，有一系列增删改查方法，通过url建立唯一标识符

标准urI写法

解释：

authority + path

authority： 包名+provider

path：表名

得到urI后，将其解析成对象传入

有两种取数方式：

  • dir：全表数据content://com.example.app.provider/table1/
  • item：单条数据content://com.example.app.provider/table1/#id

查询示例

各参数示例：

  • uri：from table – 某张表
  • projection: select column1, column2 – 所需列名
  • selection: where column = value 指定where约束条件
  • selectonArgs ： where对应的值
  • sortOrder：排序方式

通过一下方式得到逐条记录。

ContentProvider

数据的提供者，提供外部访问数据接口，依赖的就是ContentProvider

如何实现？

  1. 定义数据库表设计，url
  2. 定义对应的DbHelper
  1. 定义ContentProvider, 重写其增删改查逻辑
  1. Manifest.xml注册

解释：

context?.contentResolver?.notifyChange(uri, null) 的作用是通知所有监听了该 URI 的观察者“数据发生了变化”。它主要用于配合以下场景：

  1. Cursor 自动刷新
    如果某个地方通过 ContentResolver.query(uri, …) 得到一个 Cursor，并在查询时调用过 cursor.setNotificationUri(contentResolver, uri)，那么当 notifyChange 被触发时，该 Cursor 注册的观察者（比如 CursorAdapter、LoaderManager、ContentObserver）会收到变化通知，从而自动重新查询或刷新界面。
  2. 跨组件/跨进程同步
    其他组件或应用只要注册了针对这个 URI 的 ContentObserver，也能在数据更新时立即获知，省去轮询的开销。

简而言之：notifyChange 是 ContentProvider 的“数据变了”信号，修改/删除/插入成功后调用它，可以让使用者（UI 或其他模块）及时刷新数据。
